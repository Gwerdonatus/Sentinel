# ADR-012: actor_type as a Denormalized Field on AuditEvent

## Status
Accepted

## Date
2025-02-01

## Context
With AI agents now treated as first-class actors alongside humans and services, the audit ledger needs to distinguish actor types. The question is where this field lives: a separate `Actor` table joined to `AuditEvent`, or a denormalized field directly on `AuditEvent`.

## Decision
Add `actor_type` and `agent_name` directly to `AuditEvent` as indexed columns. No separate `Actor` table for Phase 3.

## Rationale
- Investigation queries must be fast: "show me all AI agent events in the last hour" should be a single indexed column scan, not a join.
- The audit ledger's core value proposition is rapid reconstruction during incidents. Joins add latency at exactly the moment speed matters most.
- Actor identity for humans already lives in `SentinelUser` (Phase 2) and for AI agents/services in `APIKey` (Phase 3). The audit event captures a *snapshot* of the actor at event time — consistent with the existing denormalization pattern for `actor_email` and `actor_role` from Phase 2.
- A normalized `Actor` table would require a migration to backfill historical events with synthetic actor records, adding complexity with no query benefit at current scale.

## Consequences
If actor metadata (e.g. an AI agent's description) changes after many events have been recorded, those events retain the old snapshot. This is correct behavior for an audit system — the record should reflect what was true at the time of the action, not the current state.

# ADR-013: Structured JSON Conditions Over a Custom Rule DSL

## Status
Accepted

## Date
2025-02-01

## Context
Alert rules need a condition language. Options considered: (1) a custom parsed DSL (e.g. `risk_score > 80 AND actor_type = 'AI_AGENT'`), (2) a Python `eval()`-based expression, (3) structured JSON with a fixed operator set.

## Decision
Structured JSON conditions evaluated by a recursive evaluator (`sentinel/risk/evaluator.py`). No parser, no `eval()`.

```json
{
  "operator": "AND",
  "conditions": [
    {"field": "actor_type", "operator": "eq", "value": "AI_AGENT"},
    {"field": "risk_score", "operator": "gte", "value": 50}
  ]
}
```

## Rationale
- **Security**: `eval()` on user-supplied strings is a remote code execution vector, full stop. Not acceptable for a security product.
- **No parser needed**: a custom DSL string format requires writing and maintaining a parser, with all the edge cases that implies (operator precedence, quoting, escaping). JSON parsing is already solved by the standard library.
- **Directly storable and queryable**: conditions live in a `JSONField` on `AlertRule`, trivially serializable to/from the API, and inspectable in the Django admin without a custom renderer.
- **Sufficient expressiveness**: AND/OR composition with field/operator/value triples covers every Phase 3 alerting need (single-field thresholds, compound human+AI conditions, metadata sub-field access via `metadata.amount`).

## Consequences
Complex boolean logic (e.g. NOT, XOR) is not directly supported. This is an acceptable limitation — AND/OR composition handles the threat models in scope for Phase 3. A DSL can be introduced later as a compilation target that emits this same JSON structure, with no breaking change to stored rules.

# ADR-014: Risk Score Persisted via Direct Update, Bypassing the Immutability Guard

## Status
Accepted

## Date
2025-02-01

## Context
ADR-011 established that `AuditEvent` records are immutable — the repository raises `NotImplementedError` on update. The risk engine, however, needs to write a computed `risk_score` onto each event after creation, since scoring requires historical context (other events) that doesn't exist at the moment the event itself is created.

This creates an apparent conflict: the immutability contract says "never update," but the risk pipeline needs to set one field after the fact.

## Decision
The risk service (`sentinel/risk/services.py`) writes `risk_score` directly via `AuditEvent.objects.filter(id=...).update(risk_score=...)`, deliberately bypassing the `AuditEventRepository` (which still raises on `update()`/`delete()` for all other callers).

## Rationale
- `risk_score` is a system-computed annotation, not part of the original record of *what happened*. The event's `event_type`, `actor_id`, `metadata`, and `signature` — the fields that answer "what occurred and who is accountable" — never change. `risk_score` answers a different question: "how anomalous did this look, evaluated after the fact with more context."
- The HMAC signature (ADR-011) deliberately does NOT cover `risk_score`. The signature is computed and stored at creation time over the original fields only. This means: even though `risk_score` can be updated, the cryptographic proof of the original event content is untouched. Tamper detection on the *facts* of the event remains fully intact.
- Restricting this exception narrowly — only the risk pipeline, only this one field, only via direct `update()` rather than `save()` — keeps the blast radius of the exception small and auditable. No other code path can modify audit events.

## Consequences
A compliance auditor inspecting the immutability guarantee must understand that `risk_score` is the sole documented exception, and that it is excluded from the signature scope precisely because of this. This is now explicit in code comments in both `AuditEvent` (model) and `RiskService.process_event` (the only call site), and in this ADR for anyone auditing the security model itself.
