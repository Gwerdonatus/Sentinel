# ADR-003: Celery Before Kafka

## Status

Accepted

## Date

2025-01-01

## Context

Sentinel's event processing pipeline will eventually require Kafka for high-throughput, ordered, durable event streaming. However, Kafka also requires a schema registry, consumer group management, partition topology decisions, and operational expertise to run correctly. These decisions depend on understanding the actual event schema, which does not exist yet.

## Decision

Use **Celery backed by Redis** for Phase 1 async task processing. Introduce Kafka in Phase 2 when the event schema is defined and the consumer topology can be designed correctly.

## Rationale

Introducing Kafka before the event schema exists would require either:
- (A) Designing a generic `Any` event envelope that defeats the purpose of a schema
- (B) Committing to an event schema before the product is understood

Both options are worse than deferring Kafka to Phase 2.

The task interface (`tasks/`) is abstracted. Celery tasks in Phase 1 will be migrated to Kafka producers/consumers in Phase 2 by changing the transport layer, not the business logic.

## Consequences

Phase 1 cannot replay events from a Kafka topic because there is no Kafka. This is acceptable because Phase 1 has no events to replay — the audit ledger does not exist yet.

# ADR-004: PostgreSQL Partitioning Strategy

## Status

Accepted

## Date

2025-01-01

## Context

The audit event table will receive millions of rows per day in a production deployment. Without a partitioning strategy, queries that scan historical data will degrade as the table grows. Retrofitting partitioning onto an existing table is a painful, high-risk migration.

## Decision

Design the `audit_events` table schema from day one with `created_at` as the partition key. Use **range partitioning by month**. The table is not partitioned in Phase 1 (data volume does not require it), but the schema is structured so that enabling partitioning requires only a DDL change, not an application change.

Constraints applied now:
- `id` is UUID (not serial int — serial ints cannot be used as partition keys in all scenarios and expose volume)
- `created_at` is `timestamptz` with `NOT NULL` and a database default
- No `ON DELETE CASCADE` on the audit table — audit records are immutable
- No UPDATE allowed on audit records at the application layer

## Consequences

Applications cannot rely on `id` ordering for time-ordered queries. They must use `created_at` ordering with appropriate indexes. This is correct behavior — event ordering is temporal, not insertion-sequence.

# ADR-005: UUID Primary Keys

## Status

Accepted

## Date

2025-01-01

## Context

Primary key choice affects security, scalability, and API design.

## Decision

All models use **UUID v4 primary keys** (`uuid.uuid4`).

## Rationale

- **Non-enumerable** — Sequential integers expose record counts and allow enumeration attacks (`GET /events/1`, `GET /events/2`). UUIDs make enumeration statistically impossible.
- **Distributed-safe** — UUIDs can be generated client-side or in multiple services without coordination. This is critical when Kafka producers in Phase 2 need to assign event IDs before database insertion.
- **Merge-safe** — If Sentinel ever needs to merge event data from multiple tenants or deployments, UUID collisions are negligible.

## Consequences

UUIDs are 36 characters vs 8 bytes for a bigint. Index size is larger. For the audit table, which will be the largest table in the system, this is a real cost. It is accepted because the security and distribution properties outweigh the storage overhead.

# ADR-006: OpenTelemetry From Day One

## Status

Accepted

## Date

2025-01-01

## Context

Observability is typically added after the fact, as a retrofit. This produces inconsistent trace coverage, missing spans, and gaps in the audit trail.

## Decision

Instrument Sentinel with **OpenTelemetry** before any business logic is written. Every HTTP request must carry a `trace_id` from the moment it enters the system.

This means:
- `opentelemetry-instrument` runs at Django startup
- A `RequestIDMiddleware` injects `X-Request-ID` on every request
- A `TraceContextMiddleware` propagates W3C `traceparent` headers
- All structured log lines include `trace_id`, `request_id`, `span_id`
- Celery tasks inherit the parent trace context

## Rationale

A security platform without full request tracing is a security platform that cannot investigate incidents. If a suspicious event fires an alert, the engineer investigating must be able to pull the full trace — every database query, every cache hit, every external call — for that exact request.

Adding this after the fact requires touching every view and service. Adding it now requires one middleware and one startup hook.

## Consequences

Every response includes `X-Request-ID` in headers. Logs are structured JSON (not plain text). This is a requirement for Sentinel's use case, not a trade-off.

# ADR-007: Service + Repository Pattern

## Status

Accepted

## Date

2025-01-01

## Context

Django views are frequently used as the location for business logic. This creates views that are hard to test, impossible to reuse, and tightly coupled to HTTP semantics.

## Decision

Enforce a **three-layer architecture**:

```
HTTP Request
    │
    ▼
View (thin — validates request, calls service, returns response)
    │
    ▼
Service (business logic, orchestration, no HTTP awareness)
    │
    ▼
Repository (database access, no business logic)
```

Rules:
- Views never query the database directly
- Services never import from `rest_framework` (no HTTP coupling)
- Repositories never contain business logic or validation
- Business rules live exclusively in services

## Rationale

When the risk engine needs to compute a risk score for an event, it calls `EventService.compute_risk(event_id)`. That same function is callable from a Celery task, a management command, a test, or a Kafka consumer. If the logic lived in a view, none of those callers could reach it without an HTTP request.

## Consequences

More files, more imports. A simple CRUD endpoint requires a view, a serializer, a service, and a repository. This is the correct trade-off for a platform designed to have 15 interconnected capabilities.

# ADR-008: API Versioning Strategy

## Status

Accepted

## Date

2025-01-01

## Context

Sentinel's API will be consumed by internal dashboards, external integrations, and potentially third-party SDK clients. Breaking changes to the API must not break existing clients.

## Decision

Use **URL path versioning**: `/api/v1/`, `/api/v2/`.

All endpoints live under a version prefix. The `api/` directory is organized as `api/v1/`, `api/v2/`. Version negotiation via `Accept` header is explicitly rejected.

## Rationale

URL versioning is the most operationally transparent approach. Engineers can see exactly which version a request targets by reading the URL. Logs, dashboards, and rate limit rules can filter by version prefix. Header-based versioning requires middleware to parse `Accept` headers and makes caching more complex.

## Consequences

When v2 is introduced, some code will be duplicated between `api/v1/` and `api/v2/` until v1 is sunset. This is acceptable — the alternative (shared view logic across versions) creates version coupling that is harder to untangle.

Deprecation policy: a version receives at minimum 6 months of support after a successor version is released.
