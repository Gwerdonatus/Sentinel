# ADR-009: Custom AbstractBaseUser

## Status
Accepted

## Date
2025-01-01

## Context
Django ships with a default User model that uses `username` as the primary identifier. Sentinel needs `email` as the login field, a `role` field for RBAC, and security-specific fields (`last_login_ip`, `failed_login_count`, `must_change_password`) that don't exist on the default model.

## Decision
Extend `AbstractBaseUser` with `SentinelUser`, registered as `AUTH_USER_MODEL = "sentinel_auth.SentinelUser"` before the first migration.

## Rationale
- Email is the correct identifier for a B2B security platform
- Username is unnecessary and creates an additional unique constraint with no security value
- Role as a DB column (not Django groups) is simpler and more auditable for a 4-role system
- Security fields belong on the User model itself, not a separate profile table
- `AbstractBaseUser` (not `AbstractUser`) avoids inheriting `first_name`, `last_name`, `groups` management, and `user_permissions` tables we don't use

## Consequences
**Must be set before any migration.** Changing `AUTH_USER_MODEL` after migrations exist is a multi-step, high-risk process. This is the correct permanent configuration.

# ADR-010: Redis JWT Blacklist Over DB

## Status
Accepted

## Date
2025-01-01

## Context
`djangorestframework-simplejwt` ships with a database-backed token blacklist (`rest_framework_simplejwt.token_blacklist`). Every authenticated request would require a `SELECT` on the `outstanding_tokens` + `blacklisted_tokens` tables to check validity.

## Decision
Maintain the JWT blacklist in Redis using the existing cache layer. Key pattern: `sentinel:jwt:blacklist:{jti}`. TTL set to match the refresh token lifetime so entries auto-expire.

## Rationale
- O(1) Redis lookup vs O(log n) indexed DB query
- Zero additional DB load on the authentication hot path
- Keys self-expire — no cleanup job required
- Redis is already required in the stack (Celery broker, cache layer)
- TTL-based expiry prevents unbounded growth

## Consequences
If Redis is flushed (e.g. `FLUSHALL`), all blacklisted tokens are cleared. Logged-out users could theoretically reuse refresh tokens until they expire. Mitigation: never flush production Redis; use persistence (`appendonly yes`). This is acceptable for the current risk profile.

# ADR-011: HMAC-SHA256 Audit Event Signatures

## Status
Accepted

## Date
2025-01-01

## Context
Audit records are legally and operationally significant. If a record is modified after the fact (by a malicious insider or a system bug), there must be a way to detect it.

## Decision
Compute an HMAC-SHA256 signature over key audit event fields at creation time and store it alongside the record. Signature covers: `id`, `event_type`, `actor_id`, `actor_email`, `created_at`, and SHA256 of sorted-JSON metadata.

Verification endpoint (`GET /api/v1/events/{id}/verify/`) recomputes the signature and compares with `hmac.compare_digest`.

## Rationale
- Detects post-creation tampering of any signed field
- `hmac.compare_digest` prevents timing attacks
- Metadata is hashed (not included raw) to handle large payloads efficiently
- Sort-keyed JSON serialization ensures metadata key order doesn't affect signatures
- Uses `SECRET_KEY` as the HMAC key — rotating the key means historical signatures must be re-verified with the old key (documented in runbook)

## Consequences
If `SECRET_KEY` is rotated, historical event signatures will fail verification unless the old key is retained for verification-only use. This is an acceptable operational constraint documented in the security runbook.
